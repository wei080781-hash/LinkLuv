### 5.如何確定影片夠用
1. 第一道關卡：檔案大小限制
   * upload_max_filesize: 單個檔案的最大限制 → 您設定為 50M。
   * post_max_size: 整個表單（包含文字、多個檔案）的最大限制 → 您設定為 55M。
   * 結論：只要使用者上傳的影片 ≤ 50MB，PHP 就會允許這個檔案進入伺服器。如果超過 50MB，PHP 會直接拒絕，甚至不會進入 Laravel 的程式碼。


2. 第二道關卡：上傳時間限制
這是最容易被忽略的。上傳 50MB 的影片需要時間，如果使用者的網速較慢，上傳時間可能會超過 PHP 的預設限制。
  * 檢查項：php.ini 中的 max_execution_time。
  * 風險：如果這個值是 30 (秒)，而使用者上傳 40MB 影片花了 40 秒，伺服器會在 30 秒時強制中斷連線，導致上傳失敗。
  * 建議：對於影片上傳，建議將此值調高（例如 300 秒）。

3. 第三道關卡：記憶體限制 (已足夠 ✅)
   * 檢查項：php.ini 中的 memory_limit。
   * 分析：您設定了 256M。因為 Laravel 在處理檔案上傳時，通常是先將其存為臨時檔，然後移動位置，並不會把整個 50MB 影片全部塞進記憶體裡。
   * 結論：256M 綽綽有餘，不會造成崩潰。

4. 第四道關卡：程式邏輯限制 (即將實作 🛠️)
  即使 PHP 允許 50MB，我們在 Laravel 的 MessageController 中仍然可以設定更細緻的規則。
   * 範例：我們可以在程式碼中寫：'video' => 'max:20480' (限制影片最高 20MB，即使 PHP 允許 50MB)。
   * 目的：這樣可以給使用者更友善的報錯訊息（例如：「影片太大了，請上傳 20MB 以下的檔案」），而不是讓伺服器直接報錯 500 或 413。

   * 如果您的目標是： 使用者上傳 15-30 秒的短影片（壓縮過） → 50MB 非常足夠。
   * 如果您的目標是： 使用者上傳 幾分鐘的高畫質影片 → 50MB 完全不夠，需要調高到 500M 甚至更多，並且必須大幅調高 max_execution_time。

###  6.在 $request->validate() 中加入 video 驗證規則
     1. 如何在 $request->validate() 中加入驗證規則？
        在 Laravel 的控制器（Controller）中，$request->validate() 接收一個關聯陣列 (Associative Array)。
        * 鍵 (Key)：是您 HTML 表單中 <input name="xxx"> 的名稱。
        * 值 (Value)：是您想要套用的驗證規則（用 | 符號分隔）。
         實作步驟：
        1. 找到您的 MessageController 中處理儲存（通常是 store 函式）的地方。
        2. 在 $request->validate([ ... ]) 的陣列中，新增一行針對 video 欄位的規則。

        語法結構如下：

        1 $request->validate([
        2     'content' => 'required|string|max:2000', // 既有的規則
        3     'image'   => 'nullable|image|max:5120',  // 既有的規則
        4     'video'   => 'nullable|mimes:mp4,mov,ogg|max:20480', // <--- 您要加       入的這行
        5 ]);

        規則詳細解析：
         * nullable：告訴 Laravel 「這個欄位可以不填」。如果使用者沒上傳影片，程式不會報    錯，會直接跳過。
         * mimes:mp4,mov,ogg：這是最關鍵的安全檢查。它會檢查檔案的 MIME 類型，確保上傳的是真正的影片檔。如果使用者把一個 .exe 惡意程式改成 .mp4 檔名上傳，這個規則依然      能偵測出來並攔截它。
         * max:20480：限制檔案大小。單位是 KB。
             * 計算方式：20  MB × 1024 = 20480  KB。
             * 這確保了即使 php.ini 允許 50MB，但您的業務邏輯只允許 20MB。

        ---

        2. 為什麼要這樣做？ (核心原因)

        您可能會想：「我已經在 php.ini 設定了 50MB，為什麼還要寫這行程式碼？」
        原因有三個：安全性、穩定性、用戶體驗。

        A. 防止伺服器被攻擊 (安全性 🛡️)
        如果沒有 mimes 驗證，駭客可以上傳一個偽裝成影片的 PHP 腳本。如果您的伺服器設定不    慎，駭客可能會透過訪問該檔案來執行惡意指令，直接控制您的伺服器。mimes 驗證能確保      進入伺服器的只有真正的媒體檔。

        B. 保護儲存空間與效能 (穩定性 ⚙️)
        雖然 php.ini 允許 50MB，但如果 100 個使用者同時上傳 50MB 的影片，您的硬碟空間會     迅速被填滿，且伺服器處理大檔案的壓力會增加。
        透過 max:20480，您可以精確控制每個留言能佔用的最大空間，防止資源被濫用。

        C. 提供友善的錯誤回饋 (用戶體驗 👤)
         * 沒有驗證時：使用者上傳 60MB 檔案 → PHP 直接攔截 → 瀏覽器顯示冷冰冰的 413     Payload Too Large 或 500 Server Error → 使用者覺得網站壞了。
         * 有驗證時：使用者上傳 60MB 檔案 → Laravel 攔截 → 頁面跳回原處並顯示 「The     video may not be greater than 20280 kilobytes.」 → 使用者知道該怎麼修正。

        總結
        這行代碼就像是伺服器的「安檢門」。php.ini 是大門（決定誰能進來），而    $request->validate() 是精細的安檢（決定進來的人是否攜帶違禁品、體重是否超標）。
